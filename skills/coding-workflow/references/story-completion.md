## 完成检查清单

### 开发阶段

- [ ] 需求理解清晰
- [ ] 设计方案确定
- [ ] 技术选型确定
- [ ] 开发环境准备
- [ ] **设计文档已阅读并交叉对照**（见下方"设计文档优先"规范）
- [ ] **测试骨架已创建**（TDD 第一步）

### ⚠️ 设计文档优先（强制）

> **教训来源**：Story 7.5-7 Task 6 实现时，只读了 task 简述，把"选择算法→配置参数"简化成了通用 JSON 文本框。设计文档 4.2 节明确要求 15 种算法各自的动态参数配置表单。导致漏掉功能，事后返工。

**核心规则：实现任何 Story task 前，必须先读取 Story 中 References 引用的设计文档对应章节。以设计文档的详细规格作为实现的唯一标准。**

| 步骤 | 说明 |
|------|------|
| 1. 找到设计文档 | 读取 Story 文件底部的 `References` 部分，定位相关设计文档 |
| 2. 阅读对应章节 | 读完整章节到参数级别的细节，不只是标题 |
| 3. 逐条对照 | 用设计文档的每个字段/交互/参数要求做 checklist |
| 4. 冲突处理 | 如果 task 简述和设计文档有冲突，以设计文档为准 |
| 5. 实现后验证 | 代码完成后，逐条确认设计文档中的每一项都已覆盖 |

**特别适用于前端 UI task**：
- 设计文档中的页面线框图、按钮文案、提示文字、表单布局都是强制要求，不是建议
- 参数配置表单的字段类型、校验规则、默认值必须与设计文档完全一致

### ⚠️ 编码阶段（严格遵循 TDD）

**遵循 [TDD 开发工作流](../04-testing/tdd-workflow.md) 的 Red → Green → Refactor 循环**

| TDD 步骤 | 检查项 | 说明 |
|---------|--------|------|
| **Red** | 测试先于实现 | 写测试 → 运行 → 确认失败 |
| **Green** | 最小实现通过 | 写最少代码 → 运行 → 确认通过 |
| **Refactor** | 优化代码 | 重构 → 运行 → 确认仍通过 |

**强制约束**：
- [ ] DomainService：每个方法先写测试再实现
- [ ] AppService：参数校验、DTO转换测试先于实现
- [ ] 每个方法有正常路径 + 异常路径 + 边界条件测试
- [ ] 每次 git commit 前 `mvn test` 通过

**编码阶段检查清单**：
- [ ] Entity创建完成
- [ ] Repository创建完成
- [ ] DomainService创建完成（**测试驱动**）
- [ ] AppService创建完成（**测试驱动**）
- [ ] Controller创建完成
- [ ] 代码符合规范
- [ ] 代码已格式化
- [ ] 代码已重构（测试仍通过）
- [ ] 单元测试覆盖率达标（DomainService ≥80%）

### E2E 测试阶段（如有 UI 变更）

- [ ] E2E 测试编写完成（用户旅程驱动）
- [ ] E2E 测试通过

### ⚠️ API变更测试同步（如有API变更）

**遵循 [E2E测试规范 - API变更测试同步规则](../04-testing/e2e-testing.md#八api变更测试同步规则)**

| 变更类型 | 必须更新的测试 | 检查 |
|---------|---------------|------|
| 新增端点 | E2E测试 + 后端单元测试 + 前端类型定义 | ☐ |
| 修改参数 | E2E测试 + 前端类型定义 | ☐ |
| 修改返回值 | E2E测试 + 后端单元测试 | ☐ |
| 删除端点 | 删除对应E2E测试 + 删除前端调用 | ☐ |

- [ ] 所有API变更已同步更新测试
- [ ] E2E测试覆盖新API的使用场景

### ⚠️ 状态转换门禁（测试阶段 → 审查阶段）

**Story 状态 `in-progress` → `review` 必须满足以下条件：**

| 检查项 | 要求 | 验证方式 |
|--------|------|---------|
| TDD 流程 | 测试先于实现代码编写 | 见 [TDD 开发工作流](../04-testing/tdd-workflow.md) |
| 单元测试 | 编写完成且通过 | `mvn test` 无失败 |
| 覆盖率 | DomainService ≥80%, AppService ≥65% | `mvn jacoco:report` |
| E2E测试 | 如有UI变更，测试已编写 | `npx playwright test` 通过 |

**🚫 禁止跳过测试直接进入 review**

如果尝试在无测试情况下将状态改为 `review`，应拒绝并要求补充测试。

## 代码提交

### 提交规范

```bash
# 1. 运行提交前检查脚本（推荐）
./scripts/pre-pr-check.sh

# 或手动执行以下检查：
# 1.1 确保所有测试通过
mvn test
mvn jacoco:check

# 1.2 确保E2E测试通过（如有UI变更）
cd frontend/dataengine-frontend
npm run test:e2e

# 2. 确保代码符合规范
mvn checkstyle:check
mvn spotbugs:check

# 3. 提交代码
git add .
git commit -m "feat(user): 添加用户管理功能"

# 4. 推送到远程
git push origin feature/user-management
```

### 提交前检查脚本

项目提供了 `scripts/pre-pr-check.sh` 脚本，在提交PR前自动运行所有必要检查：

```bash
# 运行检查
./scripts/pre-pr-check.sh

# 仅运行后端测试
./scripts/pre-pr-check.sh --backend

# 仅运行前端E2E测试
./scripts/pre-pr-check.sh --e2e
```

## Story关闭

### 关闭条件

Story只有在满足以下条件时才能关闭：

1. ✅ 所有验收标准已完成
2. ✅ 所有测试通过
3. ✅ 覆盖率达标
4. ✅ 代码审查通过
5. ✅ 代码已合并到develop分支
6. ✅ 文档已更新

### 关闭流程

```bash
# 1. 合并Pull Request
# 在GitHub/GitLab上点击"Merge"按钮

# 2. 关闭Issue
# 在Issue上添加评论：
"已通过Pull Request #123合并，关闭此Issue"

# 3. 更新Story状态
# 在项目管理工具（Jira/Trello）中将Story状态更新为"Done"

# 4. 归档分支
# 删除已合并的功能分支
git branch -d feature/user-management
git push origin --delete feature/user-management
```

### Story回顾

Story完成后，进行回顾总结：

```markdown
## Story回顾

### 完成情况
- 计划时间：2天
- 实际时间：2.5天
- 完成度：100%

### 遇到的问题
1. 问题1：用户名唯一性校验的性能问题
   - 解决方案：添加数据库索引
   - 耗时：0.5天

### 经验教训
1. 在设计阶段应提前考虑性能问题
2. 应提前添加数据库索引

### 改进措施
1. 在Story开始前进行技术调研
2. 在设计评审时考虑性能和安全性
```

## 最佳实践

### 1. 理解需求

在开始开发前，确保完全理解Story的需求：

- 阅读Story描述
- 与产品经理沟通
- 确认验收标准

### 2. 制定计划

制定详细的开发计划：

- 分解任务
- 估算时间
- 确定依赖

### 3. 持续集成

持续集成，及时发现和解决问题：

- 频繁提交代码
- 自动化测试
- 自动化部署

### 4. 代码审查

进行严格的代码审查：

- 自我审查
- 同行审查
- 修复问题

### 5. 文档更新

及时更新文档：

- JavaDoc
- API文档
- README
- 变更日志
