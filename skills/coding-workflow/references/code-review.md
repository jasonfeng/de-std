## 审查原则

### 代码审查的核心原则

| 原则 | 说明 |
|------|------|
| **建设性** | 提供建设性的反馈，而非批评 |
| **及时性** | 及时响应代码审查请求 |
| **全面性** | 全面检查代码的各个方面 |
| **可操作性** | 提供可操作的建议和改进方案 |
| **尊重性** | 尊重作者的意见和选择 |

### 审查目标

代码审查的目标是：
- ✅ 发现潜在的bug和问题
- ✅ 确保代码符合项目规范
- ✅ 提高代码质量和可维护性
- ✅ 分享知识和最佳实践
- ✅ 促进团队协作

## 审查流程

### 1. 提交代码审查请求

```bash
# 1. 推送代码到远程分支
git push origin feature/user-management

# 2. 创建Pull Request
# 在GitHub/GitLab上创建Pull Request
# 填写PR模板，包括：
#   - PR标题
#   - PR描述
#   - 相关Issue
#   - 测试情况
#   - 截图（如有）
```

### PR模板

```markdown
## 变更说明
简要描述本次PR的主要变更内容。

## 变更类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 重构
- [ ] 文档更新
- [ ] 其他

## 测试情况
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 覆盖率达标
- [ ] 手动测试通过

## 相关Issue
Closes #123

## 截图
（如有UI变更，请提供截图）

## 检查清单
- [ ] 代码符合项目规范
- [ ] 代码已格式化
- [ ] 已添加必要的注释
- [ ] 已更新相关文档
- [ ] 已添加必要的测试

## 审查者
@reviewer1 @reviewer2
```

### 2. 进行代码审查

```bash
# 1. 查看PR内容
# 在GitHub/GitLab上查看PR的代码变更

# 2. 拉取PR代码进行本地测试
git fetch origin pull/123/head:pr-123
git checkout pr-123

# 3. 运行测试
mvn test
mvn jacoco:check

# 4. 查看代码变更
git diff develop...pr-123
```

### 3. 提供审查反馈

```markdown
## 审查反馈

### 问题
- [ ] 代码不符合规范
- [ ] 潜在的bug
- [ ] 性能问题
- [ ] 安全问题

### 建议
- [ ] 代码改进建议
- [ ] 重构建议
- [ ] 最佳实践建议

### 赞扬
- [ ] 代码写得好
- [ ] 测试覆盖全面
- [ ] 文档清晰

## 审查结果
- [ ] 需要修改
- [ ] 通过
- [ ] 有条件通过
```

### 4. 修改和重新提交

```bash
# 1. 根据审查反馈修改代码
# ...

# 2. 提交修改
git add .
git commit -m "fix(user): 修复审查发现的问题"

# 3. 推送到同一分支
git push origin feature/user-management

# 4. 更新PR
# PR会自动更新，审查者可以查看新的代码变更
```

### 5. 合并PR

```bash
# 1. 审查通过后，合并PR
# 在GitHub/GitLab上点击"Merge"按钮

# 2. 选择合并策略
# - Merge commit（保留完整历史）
# - Squash merge（压缩为一个提交）
# - Rebase merge（线性历史）

# 3. 删除分支
# 合并后自动删除功能分支
```

## 审查清单

### 代码质量

| 检查项 | 说明 | 通过/不通过 |
|--------|------|------------|
| 代码符合项目规范 | 符合CODING_STANDARDS.md | ☐ |
| 代码已格式化 | 使用项目统一的代码格式化工具 | ☐ |
| 变量命名清晰 | 使用有意义的变量名 | ☐ |
| 方法长度合理 | 单个方法不超过50行 | ☐ |
| 类长度合理 | 单个类不超过500行 | ☐ |
| 无重复代码 | 遵循DRY原则 | ☐ |
| 无死代码 | 移除未使用的代码 | ☐ |

### 测试覆盖

| 检查项 | 说明 | 通过/不通过 |
|--------|------|------------|
| 单元测试通过 | 所有单元测试通过 | ☐ |
| 集成测试通过 | 所有集成测试通过 | ☐ |
| 覆盖率达标 | DomainService ≥80%，AppService ≥65% | ☐ |
| 测试用例完整 | 覆盖正常流程、异常流程、边界条件 | ☐ |
| 测试数据有效 | 使用有效的测试数据 | ☐ |

### 文档和注释

| 检查项 | 说明 | 通过/不通过 |
|--------|------|------------|
| JavaDoc完整 | 所有公共类、方法有JavaDoc | ☐ |
| 代码注释清晰 | 复杂逻辑有必要的注释 | ☐ |
| README更新 | 如有功能变更，更新README | ☐ |
| API文档更新 | 如有API变更，更新API文档 | ☐ |
| 变更日志更新 | 更新CHANGELOG.md | ☐ |

### 安全性

| 检查项 | 说明 | 通过/不通过 |
|--------|------|------------|
| 无SQL注入 | 使用参数化查询 | ☐ |
| 无XSS攻击 | 对用户输入进行转义 | ☐ |
| 无敏感信息泄露 | 不泄露密码、token等敏感信息 | ☐ |
| 权限控制正确 | 正确使用权限控制 | ☐ |
| 输入验证 | 对用户输入进行验证 | ☐ |

### 性能

| 检查项 | 说明 | 通过/不通过 |
|--------|------|------------|
| 无N+1查询 | 优化查询，避免N+1问题 | ☐ |
| 无内存泄漏 | 无内存泄漏问题 | ☐ |
| 无循环依赖 | 无循环依赖问题 | ☐ |
| 资源正确释放 | 正确关闭数据库连接、文件等资源 | ☐ |
| 使用缓存 | 合理使用缓存提高性能 | ☐ |

## 常见问题

### 1. 代码不符合规范

**问题描述**：代码不符合项目规范

**解决方案**：
- 指出不符合规范的地方
- 提供规范的代码示例
- 说明为什么需要符合规范

**示例反馈**：
```markdown
### 代码不符合规范
`UserDomainService.java:25` - 方法命名不符合规范

**当前代码**：
```java
public UserDO createUser(UserDTO dto) {
    // ...
}
```

**建议修改为**：
```java
public UserDTO createUser(UserDTO dto) {
    // ...
}
```

**原因**：根据规范，返回值类型应该是DTO而非DO（DO是领域对象）
```

### 2. 测试覆盖率不足

**问题描述**：测试覆盖率不足

**解决方案**：
- 指出覆盖率不足的类或方法
- 建议增加测试用例
- 提供测试用例示例

**示例反馈**：
```markdown
### 测试覆盖率不足
`UserDomainService.java` - 测试覆盖率不足，当前覆盖率65%，要求≥80%

**缺少测试的场景**：
1. 用户名为空的异常场景
2. 用户名已存在的异常场景
3. 邮箱格式不正确的异常场景

**建议添加测试用例**：
```java
@Test
@DisplayName("创建用户 - 用户名为空 - 抛异常")
void createUser_UsernameBlank_Exception() {
    UserDTO dto = new UserDTO();
    dto.setUsername("");

    assertThatThrownBy(() -> userDomainService.createUser(dto))
        .isInstanceOf(BusinessException.class)
        .hasMessage("用户名不能为空");
}
```
```

### 3. 代码可读性差

**问题描述**：代码可读性差，难以理解

**解决方案**：
- 建议拆分复杂的方法
- 建议使用有意义的变量名
- 建议添加必要的注释

**示例反馈**：
```markdown
### 代码可读性差
`UserDomainService.java:50-80` - 方法过长，逻辑复杂，难以理解

**当前代码**：
```java
public UserDTO createUser(UserDTO dto) {
    // 30行代码...
}
```

**建议重构为**：
```java
public UserDTO createUser(UserDTO dto) {
    validateUser(dto);
    checkUsernameExists(dto.getUsername());
    UserEntity entity = convertToEntity(dto);
    userRepository.insert(entity);

}

    // 验证逻辑
}

```
```

## 最佳实践

### 1. 提供建设性的反馈

```markdown
# ❌ 不好的反馈
"代码写得太烂了，重写！"

# ✅ 好的反馈
"建议将`createUser`方法拆分为多个小方法，提高代码可读性。可以参考以下结构：..."
```

### 2. 及时响应

```markdown
# 审查者应在24小时内响应代码审查请求
# 如果无法及时响应，应提前通知
```

### 3. 专注于重要的代码

```markdown
# 不要过度关注格式化、空格等小问题
# 重点关注逻辑错误、安全问题、性能问题等
```

### 4. 使用评论而非直接修改

```markdown
# 使用GitHub/GitLab的评论功能
# 不要直接修改代码再提交
```

### 5. 尊重作者

```markdown
# 尊重作者的意见和选择
# 如果有争议，可以通过讨论解决
```
