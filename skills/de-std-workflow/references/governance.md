# 规范治理机制

> 本文档定义数擎项目编码规范的治理流程，确保规范持续优化。

---

## 📋 规范文档元数据规范

所有规范文档应包含以下元数据（在文档末尾）：

```markdown
---
**文档元数据**：
- **文档ID**: CODING_STANDARDS-{category}-{name}
- **版本**: {major}.{minor}.{patch}
- **最后更新**: YYYY-MM-DD
- **维护者**: {团队/个人}
- **状态**: active | deprecated | draft
- **下次审查**: YYYY-MM-DD
---
```

### 版本号规则

| 版本类型 | 说明 | 示例 |
|---------|------|------|
| **major** | 重大变更，破坏性修改 | 1.0 → 2.0 |
| **minor** | 新增内容，向后兼容 | 1.0 → 1.1 |
| **patch** | 错误修正，小改动 | 1.0.0 → 1.0.1 |

---

## 🔄 规范变更流程

### 1. 提议变更

任何人都可以提议规范变更：

```bash
# 提交Issue
git clone <repository>
cd data-engine
gh issue create --title "规范变更：XXX" --body "变更原因和内容"
```

### 2. 讨论评审

技术架构团队评审提议：

| 评估项 | 说明 |
|--------|------|
| **必要性** | 是否确实需要变更 |
| **影响范围** | 变更影响哪些代码/文档 |
| **实施成本** | 变更实施的工作量 |
| **兼容性** | 是否破坏现有规范 |

### 3. 实验验证

对于重大变更，先进行实验：

1. 选择小范围试点
2. 收集反馈意见
3. 评估实施效果

### 4. 正式发布

变更通过后：

1. 更新规范文档版本号
2. 更新文档元数据
3. 提交PR合并
4. 通知团队成员

## 📅 定期审查机制

### 季度审查

每季度对所有规范进行审查：

| 审查项 | 内容 |
|--------|------|
| **准确性** | 规范是否与实际代码一致 |
| **完整性** | 是否有遗漏的重要规范 |
| **时效性** | 是否有过时的内容 |
| **可用性** | 是否清晰易懂 |

### 审查流程

```bash
# 1. 技术架构团队发起审查
# 2. 各领域负责人提交更新建议
# 3. 会议讨论确定更新内容
# 4. 更新规范文档
# 5. 通知团队变更
```

## 🔧 自动化检查

### 合规性检查脚本

项目提供自动化检查脚本：

```bash
# 运行合规检查
./scripts/check-compliance.sh
```

检查内容：
- Entity类是否带Entity后缀
- 是否使用禁止的SQL注解
- 是否使用@Autowired
- 主键是否使用雪花算法
- Vue组件是否使用script setup
- TypeScript是否启用严格模式
- Flyway迁移脚本是否存在
- 是否存在MySQL语法

### CI集成

将合规检查集成到CI流水线：

```yaml
# .github/workflows/coding-standards-check.yml
name: Coding Standards Check
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Check Compliance
        run: ./scripts/check-compliance.sh
```

## 📊 规范执行指标

### 代码审查通过率

| 层级 | 目标通过率 | 说明 |
|------|-----------|------|
| **核心规则** | 100% | 10条核心规则必须100%遵守 |
| **重要规范** | ≥95% | 领域核心规范 |
| **一般规范** | ≥90% | 编码风格等 |

### 合规性趋势

每月统计合规性指标：

```bash
# 查看合规性报告
./scripts/check-compliance.sh --report
```

## 🚨 规范冲突处理

### 冲突类型

| 冲突类型 | 处理方式 |
|---------|---------|
| **新旧规范冲突** | 以新规范为准，更新旧代码 |
| **跨领域冲突** | 技术架构团队协调解决 |
| **工具与规范冲突** | 以规范为准，调整工具配置 |

### 冲突上报

```bash
# 上报规范冲突
gh issue create \
  --title "规范冲突：XXX" \
  --label "conflict" \
  --body "冲突描述和重现步骤"
```
